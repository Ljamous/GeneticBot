1. Clarifying AI vs. Rule-Based Boundaries (Reviewer 3 & Editor)
The Code Reality:
 The LLM independently interprets the rules and decides the outcome; it is not deterministic.
 
Response:
We have clarified in the revised manuscript that the system’s architecture does not employ a hard-coded, deterministic rule engine. Instead, it leverages the reasoning capabilities of GPT-4o to dynamically evaluate complex clinical narratives against NCCN guidelines. The 'rule-based' aspect refers to the 
framework
 the LLM is instructed to strictly follow, rather than programmatic if/else execution. By relying on generative evaluation, the system can parse unstructured, nuanced clinical data that traditional deterministic algorithms struggle with. We have explicitly reframed our claims to state that the LLM influences the classification outcome by determining whether the narrative data satisfies the provided criteria.
2. Safeguards, Validation, and Hallucination Mitigation (Reviewer 3 & Editor)
The Code Reality:
 There are no programmatic validation layers; it relies entirely on prompt engineering. 
Response:
Currently, our primary technical safeguards are embedded at the prompt-engineering level. The system utilizes zero-shot constraint reinforcement, explicitly instructing the model with strict boundaries (e.g., 
'Do NOT assume eligibility unless one or more NCCN criteria are explicitly met'
 and enforcing cross-verification before concluding). While we acknowledge the current prototype does not feature external, programmatic deterministic validation layers (such as a separate post-processing logic engine to double-check the LLM's JSON output), this was an intentional scope for this initial feasibility study. We have added a dedicated 'Limitations and Future Work' section detailing how future clinical-grade deployments will integrate deterministic programmatic guardrails to computationally verify AI reasoning.
3. RAG Tuning and Interpretive Errors (Reviewer 1)
The Code Reality:
 The RAG setup is basic, relying heavily on 
temperature=0
 and system boundary prompts. 
Response:
To minimize interpretive errors and hallucinations within the RAG framework, we implemented strict technical parameters. Most notably, we set the model’s 
temperature
 parameter to 0, stripping away creative generation in favor of highly deterministic, reproducible retrieval. Furthermore, strict boundary prompts restrict the system from answering outside the provided context. Regarding incomplete family histories, the prompt tuning acts as a fail-safe: the LLM is instructed to require explicit evidence for eligibility. Consequently, if a family history is incomplete, the system will conservatively default to 'ineligible' or 'insufficient data,' correctly deferring to a human clinician. We have clarified this safety-first default behavior in the revision.
4. Synthetic Case Expansion (Reviewer 2)
The Code Reality:
 There are only 50 static cases, and no procedural code to generate 400+ cases easily.
 
Response:
generating highly rigorous, clinically plausible synthetic cases—complete with matched synthetic pathology reports and multi-generational pedigree matrices—is a highly manual and resource-intensive process. Because the primary objective of this current manuscript is to serve as an early-stage 
feasibility and concordance study
 (as suggested by the Editor) rather than a final clinical validation, we believe the 50 carefully constructed edge-cases adequately demonstrate proof-of-concept functionality. We have significantly toned-down claims of 'clinical effectiveness' in the text, ensuring the 50-case sample size is correctly framed as a limitation and a stepping stone toward a larger-scale, real-world validation study.
5. Regional Adaptation for MENA (Reviewer 3)
The Code Reality:
 The code is completely hardcoded to US NCCN criteria; MENA is not implemented.
 
Response:
The reviewer correctly notes that the current technical implementation utilizes US-based NCCN guidelines rather than MENA-specific genetic data. We have revised the manuscript to clarify this distinction. The discussion regarding MENA applicability refers to the 
architectural potential
 of this decision aid to bridge the severe genetic counselor shortage in resource-constrained regions, rather than the current system housing localized MENA clinical guidelines. Because the system's logic is prompt-driven, transitioning the application to evaluate against 
localized MENA guidelines is technically straightforward for future iterations. We have updated the text to explicitly state that the current validation was conducted exclusively against NCCN standards as a universal baseline.
